# LazyVim Audit Report: Katas 115-170

## 1. Batch Scope and Notable Existing Coverage

Scope inspected: `115_*.md` through `170_*.md` only, compared against `lazyvim-kata-audit/01_lazyvim_workflow_digest.md`.

This batch is strong on portable Vim fundamentals:

- File/window motion and viewport navigation: `115`-`124`
- Structural code/prose motion: `125`-`130`
- Tags and file-under-cursor workflows: `131`-`136`
- Text objects, registers, puts, substitutions: `137`-`145`
- Windows, buffers, folds, formatting, visual increment: `146`-`150`, `166`-`170`
- Quickfix/location/diagnostic bridge workflows: `151`-`160`
- Diff, terminal, shell output: `161`-`165`

Best LazyVim-adjacent coverage already present: quickfix/location workflows, Neovim diagnostics-to-quickfix, terminal mode, folds, buffer/window operations, built-in formatting/indent fallback, and tag-definition navigation.

## 3. Existing Katas That Should Remain Vanilla Vim

- `060_adjacent_line_first_non_blank.md`, `061_count_relative_first_non_blank.md`, `062_start_or_numbered_line.md`, `063_end_or_numbered_line.md`, `064_percentage_jump.md`, `120_byte_offset_jump.md`: pure motion grammar; LazyVim should not change the required behavior.
- `121_viewport_relative_landing.md`, `065_fine_viewport_scroll.md`, `066_half_page_scroll.md`, `067_full_page_scroll.md`: viewport mechanics are foundational and plugin-independent.
- `068_sentence_motion.md`, `074_sentence_text_objects.md`: built-in sentence motions/text objects should stay as prose-editing fundamentals; `mini.ai` belongs in separate code-object variants.
- `129_reverse_match_cycle.md`: keep focused on standard `matchit`; LazyVim compatibility should be only a readiness check.
- `075_recover_small_deletion.md`, `076_walk_deletion_history.md`: register behavior is core Vim state and should not be abstracted behind plugins.
- `149_restore_entire_changed_line.md`: `U` is intentionally narrow historical Vim behavior; LazyVim should not broaden the lesson.
- `165_read_shell_output_into_buffer.md`: `:read !{cmd}` is a portable shell bridge; LazyVim terminal features are adjacent, not replacements.
- `086_navigate_between_folds.md`, `087_adjust_fold_depth_incrementally.md`: fold commands should remain deterministic with marker/manual folds; LazyVim Treesitter folds can be noted separately.
- `089_generate_incrementing_sequence.md`: Visual `g<C-a>` is core editing grammar and does not need a LazyVim variant.

## 5. Proposed Next LazyVim Katas to Add

1. Flash search-to-edit loop: `s`, operator-pending `r`, repeat with `.`, with `/` fallback.
2. `mini.ai` code text objects: change function body, call args, class block; stock text-object fallback where possible.
3. LSP navigation and edit loop: `gd`, `gr`, `K`, `<leader>cr`, `<leader>ca`, `<leader>co`.
4. Diagnostics with Trouble: `[d`/`]d`, `<leader>cd`, `<leader>xx`, `<leader>xq`, fallback quickfix.
5. Gitsigns hunk workflow: `[h`/`]h`, preview, stage/reset hunk, blame, `ih`.
6. Snacks picker project workflow: find file, grep, buffers, recent files, search word/selection.
7. Formatting/linting workflow: `<leader>cf`, `<leader>cF`, autoformat toggles, built-in `=` fallback.
8. Completion/snippet workflow: engine-neutral accept/select/expand/jump, with Blink notes guarded by readiness checks.
9. Neotest workflow: nearest test, file test, last test, output, summary, CLI fallback.
10. DAP workflow: breakpoint, continue, step controls, UI, with explicit extra/readiness requirements.
